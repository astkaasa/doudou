# 城市与机场系统设计

- 状态：已实现并验收 v2（阶段 0—5 完成）
- 日期：2026-07-10
- 适用范围：城市数据、机场数据、航线模型、经济计算、事件、存档、地图与经营 UI

## 1. 文档目的

本方案用于指导城市数据重整和机场系统的分阶段实现。它首先锁定数据边界、稳定标识、历史口径和存档兼容方式，再允许具体数值通过数据审计与平衡模拟逐步校准。

本方案按独立可验收的阶段实施。阶段 0—5 已全部落地；后续仍延后的时刻交易、同城市对多航线和通勤航空不属于本轮完成条件。

## 2. 实施前背景

实施前，游戏仅以城市作为航线端点：

- 城市同时承载人口、商务、旅游、区域、航线距离、着陆费和投资成本。
- 航线使用无方向城市对作为唯一键。
- 地图只绘制城市点。
- 机场建设是城市级投资，为同城航线提供着陆费折扣。
- 时代覆盖 1960—2020，但没有机场启用、关闭和迁移历史。

这种模型足以支持基础经营，但会把市场规模、机场设施、机场费用和枢纽能力混成一个城市等级，也无法表达同城多机场、跑道限制、次级机场、机场迁移或备降。

同时，当时的城市数据存在需要优先修正的问题：

- 人口口径混用行政市、城区和都会区。
- subRegion 同时承担增长、事件和网络分组，造成地理误分。
- 部分名称不是城市名称，如“文莱”和“冲绳”。
- 部分首都字段与坐标需要修正。
- level 同时影响需求、费用和投资，无法表达“大市场、弱设施”或“小市场、强枢纽”。

## 3. 目标

### 3.1 核心目标

1. 将“城市市场”和“机场运营节点”拆分为独立领域对象。
2. 使用可追溯来源重整 1960、1975、2000 三个时代锚点的城市市场数据。
3. 保持现有城市 ID 稳定，不因显示名、区域或机场变化破坏事件和存档。
4. 允许一个城市拥有一座或多座机场。
5. 让机场选择产生明确经营取舍，而不是只增加百科字段。
6. 保证数据源更新不会直接改变玩家存档或游戏平衡。
7. 继续支持离线单文件产物，不在运行时联网获取数据。

### 3.2 非目标

首批实现不包含：

- 逐小时航班计划、昼夜或真实时刻表。
- 真实世界逐项机场收费单。
- 完整空管、频率、导航台或跑道方向模拟。
- 将 OurAirports 全量设施显示在世界地图。
- 把私人小机场自动视为可供定期客运使用。
- 一次性实现所有历史机场和所有高级机场玩法。
- 引入 React、Vue、TypeScript、数据库或新的构建系统。

## 4. 已锁定的设计决策

以下决策在实现阶段不应随意改变；如需调整，应先更新本文件。

1. 城市代表航空市场，机场代表实际运营节点。
2. 现有城市 ID 不删除、不复用、不改义。
3. IATA、ICAO 和 OurAirports ident 不作为游戏存档主键。
4. 航线继续保留 from 和 to 城市 ID，并增加机场 ID。
5. 在允许同一城市对存在多条机场组合航线前，航线必须先拥有稳定 uid。
6. 城市人口采用航空市场或城市聚落口径，不再混用行政市人口。
7. 客观来源字段与游戏平衡字段分开保存。
8. 大型外部数据只在显式同步命令中读取；普通构建不联网。
9. 地图继续以城市为主要交互点，机场在选中城市后展开。
10. 未验证的城市—年代组合使用稳定的抽象机场回退，不伪造历史事实。
11. 旧存档中的城市、航线、总部、分部和机场投资必须可迁移。
12. 玩家可见规则变化必须更新 CHANGELOG.md，并通过 npm run check。

## 5. 领域模型

~~~mermaid
flowchart LR
  C["城市市场 City"] --> M["人口 / 商务 / 旅游需求"]
  C --> A1["机场 Airport A"]
  C --> A2["机场 Airport B"]
  A1 --> R["航线 Route"]
  A2 --> R
  P["飞机 Plane"] --> R
  E["事件与修正器"] --> C
  E --> A1
  I["机场投资"] --> A1
~~~

### 5.1 城市

城市负责：

- 市场人口。
- 商务和旅游需求。
- 网络区域与主线覆盖。
- 城市级盛事和新闻。
- 航空市场随年代的成长。
- 机场客源的共同上限。

城市不再直接负责：

- 跑道能力。
- 机场设施等级。
- 具体起降费。
- 机场容量或时刻资源。
- 机场启用与关闭。

建议的城市结构：

~~~js
{
  id: "london",
  name: "伦敦",
  lat: 51.5,
  lon: -0.11,
  region: "europe",
  networkRegion: "europe",
  eventZones: ["western_europe", "north_atlantic"],
  marketRole: "core",
  canBeHQ: true,
  capitalPeriods: [
    { fromYear: null, toYear: null }
  ],
  sourceRefs: {
    population: "un-wup-2025",
    coordinates: "verified"
  }
}
~~~

marketRole 允许以下值：

| 值 | 含义 | 默认开线 |
| --- | --- | --- |
| core | 常驻核心市场 | 是 |
| regional | 区域市场 | 是，可在 UI 折叠 |
| remote | 偏远或海岛市场 | 可通过条件或合同 |
| event | 盛事驱动市场 | 盛事期间解锁 |
| special | 特殊客源市场 | 按专属规则 |

### 5.2 城市市场状态

当前 cityStates 的 pop、biz、tour 继续作为运行时状态，但其来源和含义需要明确：

~~~js
{
  pop: 9.2,
  biz: 72,
  tour: 68
}
~~~

- pop：以百万人为单位的航空市场或城市聚落人口。
- biz：0—100 的游戏商务需求指数。
- tour：0—100 的游戏旅游需求指数。

人口是有来源的客观数据；biz 和 tour 是可解释、可复现的游戏参数，不宣称为现实统计值。

### 5.3 机场

机场负责：

- 真实或抽象坐标。
- 机场代码和名称。
- 跑道、海拔、表面和照明。
- 启用、关闭、迁移和历史名称。
- 设施、费用、容量和机场角色。
- 与城市中心的交通可达性。
- 机场级事件、投资和备降关系。

建议的机场结构：

~~~js
{
  id: "london-heathrow",
  cityId: "london",
  name: "伦敦希思罗机场",
  lat: 51.4706,
  lon: -0.4619,
  source: {
    provider: "ourairports",
    sourceId: 2434,
    ident: "EGLL",
    snapshot: "2026-07-09"
  },
  codes: {
    iata: "LHR",
    icao: "EGLL"
  },
  factual: {
    type: "large_airport",
    elevationFt: 83,
    scheduledService: true,
    maxRunwayM: 3902,
    hardSurface: true,
    lighted: true
  },
  history: [
    {
      fromYear: 1960,
      toYear: null,
      active: true,
      sourceRefs: []
    }
  ],
  gameplay: {
    role: "primary_hub",
    feeTier: 5,
    capacityTier: 5,
    accessTier: 5,
    infrastructureTier: 5
  },
  audit: {
    confidence: "verified",
    sourceRefs: []
  }
}
~~~

上述 gameplay 字段为游戏参数，不从 OurAirports type 直接等同推导。

### 5.4 抽象机场

每座城市都必须有稳定的抽象机场：

~~~js
{
  id: "virtual-london",
  cityId: "london",
  name: "伦敦城市机场",
  lat: 51.5,
  lon: -0.11,
  source: {
    provider: "abstract"
  }
}
~~~

抽象机场用于：

- 旧存档迁移。
- 尚未核实的历史年代。
- 自动匹配低置信度城市。
- 数据源暂时缺失。

抽象机场不能提供特殊费用、容量或枢纽优势。后续核实真实机场时，不能静默改变已保存航线；应通过显式迁移或机场迁移事件处理。

### 5.5 航线

目标结构：

~~~js
{
  uid: 101,
  from: "london",
  to: "newyork",
  fromAirportId: "london-heathrow",
  toAirportId: "newyork-jfk",
  assignedPlanes: [12],
  price: 720,
  suggestedPrice: 700,
  serviceMultiplier: 1,
  suspended: false
}
~~~

约束：

- from 和 to 永远是城市 ID。
- fromAirportId 和 toAirportId 必须属于对应城市，或是允许服务该城市的特殊机场。
- 机场关闭不自动删除航线。
- 在 route uid 完成迁移前，仍只允许一个城市对一条航线。
- 允许多机场并行航线后，竞争同时计算城市市场竞争与机场对重叠。

### 5.6 飞机

飞机需要新增独立机场适配字段，不能仅从现有 type 推断：

~~~js
{
  minRunwayM: 2200,
  requiredInfrastructureTier: 3,
  hotHighPerformance: 1
}
~~~

具体数值需要按机型资料与游戏平衡校准。缺少资料时使用机型类别的保守默认值，并在数据审计报告中标记。

## 6. 数据来源与可追溯性

### 6.1 城市历史人口

首选来源：

- UN DESA World Urbanization Prospects 2025
- https://population.un.org/wup/
- https://population.un.org/wup/downloads?tab=Urban+Agglomerations

该数据提供 1950—2050 年年度城市、城镇和城市聚落估算，可直接生成 1960、1975、2000 和 2020 的人口锚点。

处理规则：

1. 优先匹配统一定义的城市聚落。
2. 保存 UN 的来源标识、名称和国家信息。
3. 对名称变化或多中心都会区使用显式映射。
4. UN 数据未覆盖的小型市场，再使用国家统计机构或地方官方资料。
5. 不把行政市人口与都会区人口混合进同一字段。

### 6.2 机场和跑道

首选来源：

- https://ourairports.com/data/
- https://ourairports.com/help/data-dictionary.html
- https://github.com/davidmegginson/ourairports-data

使用 airports.csv 和 runways.csv。countries.csv 与 regions.csv 仅用于解析和校验。

暂不使用：

- airport-frequencies.csv
- navaids.csv
- airport-comments.csv

OurAirports 的 scheduled_service 表示当前状态，不能直接证明历史年代状态；small_airport 也不等于私人机场。

### 6.3 历史机场资料

来源优先级：

1. 国家民航机构、AIP、机场运营方和机场官网。
2. 政府档案、城市或交通机构。
3. 可交叉核对的历史资料。
4. 百科资料只作为线索，不单独支撑关键玩法限制。

每条历史记录至少保存：

- 来源 URL。
- 核实日期。
- 启用或关闭年份。
- 证据说明。
- 置信度。

### 6.4 游戏派生参数

以下字段属于游戏设计，不伪装为客观事实：

- biz 和 tour。
- feeTier。
- capacityTier。
- accessTier。
- infrastructureTier。
- 机场角色。
- 合同补贴、时刻价格和投资回报。

这些字段必须有可解释的生成或调参规则，并通过平衡模拟验证。

## 8. 城市数据重整

### 8.1 人口口径

事实人口与游戏需求分开：

~~~js
{
  populationM: 12.4,
  marketPopulationScore: 5.1
}
~~~

当前经济模型近似线性消费人口。改用统一都会区人口后，大型城市会过度支配需求，因此人口需求必须使用凹函数压缩。

建议初始形式：

~~~js
populationComponent = scale * Math.pow(Math.max(populationM, 0), exponent)
~~~

其中 exponent 必须小于 1。初始调参区间为 0.55—0.75，最终值以五总部完整平衡矩阵为准。

要求：

- UI 展示事实人口。
- 需求计算使用压缩后的市场值。
- 不再使用 level 补偿人口口径差异。

### 8.2 城市增长

当前人口按 subRegion 统一增长。目标改为：

- 人口增长优先使用城市级历史序列。
- 1960、1975、2000 为剧本初始锚点。
- 剧本进行中的年度增长由相邻历史锚点插值或城市级增长率产生。
- 随机波动只能小幅改变季度结果，不能长期偏离历史趋势。
- biz 和 tour 可继续使用区域趋势叠加城市标签。

### 8.3 区域职责拆分

新增或调整：

- networkRegion：主线覆盖、跨区域收益。
- eventZones：新闻、灾害和盛事作用域，可为数组。
- growthProfile：商务和旅游增长档案。

不再让一个 subRegion 同时决定全部行为。

## 9. 城市—机场自动匹配

### 9.1 候选筛选

对每座城市选取一定半径内的机场候选，并考虑：

- scheduled_service。
- large_airport、medium_airport 或有意义的 small_airport。
- IATA、ICAO 和 ident。
- municipality、name 和 keywords 匹配。
- 最长开放跑道。
- 与城市中心距离。

机场匹配不得只依赖 municipality，因为它表示主要服务城市，不一定是物理所在地。

### 9.2 评分与置信度

评分组成：

~~~text
定期航班状态
+ 机场规模
+ 有效代码
+ 可用跑道
+ 名称和服务城市匹配
- 地面距离
- 数据缺失和关闭状态
~~~

输出审计字段：

- cityId。
- 自动首选机场。
- 距离。
- 得分。
- 第一与第二候选得分差。
- 置信度。
- 警告。
- 候选列表。

置信度：

| 等级 | 行为 |
| --- | --- |
| verified | 已有来源和人工核实，可进入历史玩法 |
| high | 自动选择可靠，可作为现代默认机场 |
| medium | 进入候选列表，不自动替代历史机场 |
| low | 使用抽象机场 |

### 9.3 多机场城市

首批垂直切片覆盖：

- 伦敦。
- 东京。
- 纽约。
- 北京。
- 上海。

在此基础上又完成首尔、台北、大阪、曼谷、新加坡、巴黎和达拉斯的人工核验与历史主机场规则。这些城市共同验证：

- 主机场与次级机场选择。
- 同城机场距离和地图表现。
- 机场选择 UI。
- 跑道和机型兼容。
- 城市需求共享。
- 旧航线迁移。

## 10. 机场经济与玩法

### 10.1 市场需求与机场吸引力

城市对先生成总市场需求，机场组合只分配和修正该需求，不能为每个机场对重复生成完整市场。

建议模型：

~~~text
城市对总需求
× 机场可达性
× 机场角色适配
× 价格、品牌、竞争和事件
= 机场组合可获得需求
~~~

如果未来允许同一城市对多条机场组合航线，应按各组合吸引力分摊总需求，避免双重计算。

### 10.2 机场角色

| 角色 | 需求 | 费用 | 容量 | 典型策略 |
| --- | --- | --- | --- | --- |
| primary_hub | 高 | 高 | 易拥堵 | 传统航空、商务和转机 |
| secondary | 中 | 较低 | 较宽松 | 廉航和旅游 |
| regional | 较低 | 低 | 充足 | 支线和合同 |
| remote | 特殊 | 不定 | 有限 | 海岛、偏远和补贴 |
| special | 专属规则 | 不定 | 不定 | 朝觐、盛事或特殊任务 |

角色是游戏参数，应由事实信息和设计目标共同决定。

### 10.3 跑道和高原性能

首版优先使用客观数据形成玩法：

- 最低跑道长度。
- 硬化表面要求。
- 机场设施等级。
- 海拔与高温性能惩罚。

短跑道或高原机场优先采用减载，而不是一律硬禁止：

~~~text
可用座位 = 标准座位 × 性能系数
~~~

只有明显不满足最低安全档位时才禁止开线。

### 10.4 航程

分开两种距离：

- marketDistance：城市中心距离，用于市场和历史兼容。
- operatingDistance：具体机场距离，用于航程、燃油和地图弧线。

首批可以保持建议票价和需求使用 marketDistance，燃油与航程逐步切换到 operatingDistance，避免一次性改变全部平衡。

### 10.5 机场费用

机场费用由 feeTier、机场角色、航程、服务强度和投资共同决定。

首版不追求真实收费单，保留现有城市着陆费基线，再加入机场档位系数。所有系数必须通过模拟调校。

### 10.6 容量与时刻

容量按季度抽象为容量点，不引入小时：

~~~text
容量消耗 = 距离班次倍率 × 飞机规模倍率
~~~

短途高频航线消耗更多机场容量，长途低频航线消耗较少。

容量不足时优先产生：

- 费用上升。
- 延误或运营效率损失。
- 新航线开通成本提高。

不建议一开始简单禁止全部扩张。

时刻买卖和竞拍属于后续阶段，必须在 AI 能合理参与后才启用。

### 10.7 枢纽协同

同一机场连接多个区域时获得有上限的转机需求。计算应基于：

- 活跃航线数。
- 覆盖区域数。
- 服务强度。
- 机场容量余量。

不引入真实航班时刻或转机波次。

### 10.8 航线开发合同

区域、偏远和次级机场可以发布季度合同：

- 开航补贴。
- 起降费减免。
- 保底收入。
- 品牌或机场关系奖励。

合同要求：

- 连续运营若干季度。
- 最低服务强度。
- 可靠性或客座率目标。

违约后退还补贴或损失机场关系。合同用于利用更多小机场，而不是把它们全部放入常驻城市列表。

### 10.9 历史机场迁移

机场启用或关闭时生成明确事件：

- 提前迁入新机场可获得优惠和时刻。
- 留在旧机场可保持当前客流但承受容量压力。
- 旧机场关闭前必须迁移或关闭相关航线。
- 路线迁移保留历史指标，并支付一次性搬迁成本。

迁移状态必须持久化，不能每次载入根据年份重复弹出。

### 10.10 备降与韧性

机场级灾害不再只能被动停飞。可选处理：

- 同城备降。
- 邻近城市备降。
- 取消航班。
- 支付额外成本维持服务。

机场投资可提高除雪、防洪、备用跑道或应急保障等级。

## 11. 机场投资

城市级旧机场投资已迁移为具体机场资产，无已核实机场的存档才使用抽象机场回退。

迁移规则：

1. 旧存档中的 airport 子公司继续存在。
2. 城市只有一座已核实基地机场时，迁移到该机场。
3. 城市有多座机场时，默认迁移到当年主机场，并记录 migratedFromCity。
4. 无已核实机场时继续保持城市级折扣。
5. 折扣数值不能因迁移降低。

已实现的机场投资分支：

- 跑道扩建。
- 航站楼。
- 维修机库。
- 货运中心。
- 地面交通。
- 灾害韧性。

每座机场的升级槽位有限，避免所有机场最终同质化。

## 12. 事件和修正器

现有 activeModifiers 继续作为统一事件影响路径。

新增 scope：

~~~js
{
  kind: "airportIds",
  airportIds: ["london-heathrow"]
}
~~~

以及需要时的：

~~~js
{
  kind: "eventZones",
  eventZones: ["west_africa"]
}
~~~

匹配规则：

- 城市事件匹配 route.from 和 route.to。
- 机场事件匹配 fromAirportId 和 toAirportId。
- 区域灾害优先使用 eventZones。
- 盛事仍以城市市场为中心，可向同城机场和区域枢纽溢出。

旧 route 上不得重新加入临时需求、费用或停飞字段。

## 13. 存档兼容

### 13.1 城市

- 城市 ID 永久保留。
- 更名只修改显示名。
- 区域和市场角色变化不删除旧状态。
- 新城市通过 normalizeCityStates 自动补齐市场数据。
- 被降级城市上的总部、分部和航线继续有效。

### 13.2 航线

迁移顺序：

1. 为所有旧航线分配稳定 uid。
2. 保留 from 和 to。
3. 根据存档年份选择已核实默认机场。
4. 无法核实时分配抽象机场。
5. 重新计算建议票价和运营指标时保留上季度展示值。

### 13.3 机场关闭

机场关闭不能在载入时直接删除航线。状态统一进入：

~~~text
active -> relocation_pending -> relocated / closed
~~~

relocation_pending 时允许查看财报和存档，但推进季度前必须处理受影响基地或航线。

### 13.4 新版本存档

实现每个破坏性结构变化时：

- 提升 SAVE_VERSION。
- 添加显式迁移函数。
- 补充旧版本、损坏字段和重复字段测试。
- 新版本字段加入 PERSISTED_GAME_STATE_FIELDS。

## 14. UI 与地图

### 14.1 地图

- 默认只显示城市点。
- 城市点击、hover、总部和航线高亮保持现有行为。
- 选中城市后在城市面板展示机场卡片。
- 放大到合适级别时才可显示机场小点。
- 航线弧线使用具体机场坐标，但城市仍是主要触控热区。
- 不把 OurAirports 全量设施加入 DOM 或 SVG。

### 14.2 城市选择

城市列表默认展示 core 和 regional：

- remote、event 和 special 使用独立分组。
- 盛事或合同激活时自动提升对应城市。
- 搜索结果可以包含当前不可常规开线的城市，并解释解锁条件。

### 14.3 开航线流程

建议流程：

~~~text
选择起飞基地城市
-> 选择目的城市
-> 如有必要选择机场
-> 选择飞机
-> 设置票价
-> 查看完整预估
~~~

城市只有一个可用机场时自动选择，不增加一次点击。

预估至少显示：

- 机场名称和代码。
- 城市距离与运营距离。
- 跑道兼容或减载。
- 机场费用影响。
- 容量余量。
- 客座率、利润和开通后现金。

### 14.4 航线管理

航线表增加机场代码，但移动端优先显示城市名称：

~~~text
东京 HND -> 伦敦 LHR
~~~

机场迁移、减载、容量不足和合同状态使用可读文字，不能只依赖颜色或图标。

## 17. 测试与验收

### 17.1 数据测试

- 城市 ID 唯一且稳定。
- 城市经纬度有效。
- 新城市均有三个时代市场锚点。
- 每座城市至少有抽象机场。
- 机场 ID 唯一。
- 机场 sourceId 与 ident 关联有效。
- 跑道聚合忽略关闭跑道。
- 每个城市—年代至少有一座可用机场。
- 低置信度匹配不会自动进入历史玩法。

### 17.2 领域测试

- 市场需求不会因多个机场重复计算。
- 非默认人口压缩参数行为稳定。
- 跑道不足、减载、零座位和缺失数据。
- 高原性能与不同飞机字段。
- 机场距离和城市距离分别使用。
- 机场费用与投资折扣只计算一次。
- 同机场和不同机场竞争。
- 机场级事件与城市级事件叠加。
- 停飞、换机、迁移和关闭。

### 17.3 存档测试

- 无机场字段的旧航线。
- 旧城市级机场投资。
- 被降级城市作为总部或分部。
- 新城市市场补齐。
- 不存在或已关闭的机场。
- 重复 route uid 和无效机场引用。
- pending 机场迁移只触发一次。

### 17.4 UI 与 E2E

桌面和手机至少验证：

- 选择时代和总部。
- 选择城市与机场。
- 购买飞机。
- 因跑道不足被正确提示。
- 开通航线并推进季度。
- 航线管理和机场代码显示。
- 存档、读档与迁移。
- 城市点、机场点和航线仍可触控。

### 17.5 平衡验收

任何影响需求、费用、航程或容量的阶段都必须运行：

~~~bash
npm run check
npm run balance:acceptance -- --workers 8 --strict
~~~

继续使用北京、迪拜、伦敦、纽约和悉尼五总部矩阵，并在新增城市允许作为总部后增加专项样本。

验收要求：

- 原有四时代指标仍落在 docs/balance-targets.md 区间。
- 不出现所有最优航线都集中于少数超大城市。
- 不出现某区域总部因机场兼容而必败。
- 机场非航线收入不超过既有非航线收入限制。
- 多机场选择至少存在两种可行策略，而不是唯一最优机场。

## 18. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 真实人口使大城市需求失控 | 使用凹函数压缩并跑完整平衡矩阵 |
| 当前机场被错误用于早期年代 | 历史核实或抽象机场回退 |
| 多机场重复生成需求 | 先生成城市对总需求，再分摊 |
| 机场代码变化破坏存档 | 使用游戏稳定 ID |
| 全量机场拖慢地图 | 城市主交互、按需显示机场 |
| 区域重分破坏事件 | 拆分 eventZones 并补事件测试 |
| 旧机场投资价值下降 | 迁移时保留或提高原折扣 |
| 数据同步改变平衡 | 固定快照，人工审查生成差异 |
| 玩法一次扩张过大 | 严格按阶段交付 |

## 19. 延后决策

### 19.1 阶段 4 已决策

- 容量按季度点数计算，capacityTier 1—5 的基础容量依次为 12、20、32、48、72 点。
- 玩家航线按完整容量计入，AI 航线按 50% 折算为背景拥堵；容量不足先增加费用、降低需求效率并提高新航线开通成本，不直接封锁全部扩张。
- 枢纽协同要求同一机场至少 3 条活跃航线和 2 个 networkRegion，并按容量余量衰减，需求奖励上限 12%。
- 机场关系范围为 -100—100；正关系最多降低 5% 机场费，合同达标提升关系，违约降低关系。
- 每座已投资机场固定 3 个升级槽位，每个分支最多升级一次。分支为跑道、航站楼、维修机库、货运中心、地面交通和灾害韧性。
- 普通开线只展示 high / verified 机场；candidate 机场只有在距离、跑道和设施过滤通过后，才可通过季度机会合同进入玩家路径。
- 航线开发合同提供开航补贴、季度保底、起降费减免和完成奖励，要求连续运营、最低服务强度和客座率；无法完成时返还部分补贴并损失机场关系。

### 19.2 阶段 5 已决策

- 历史机场使用按年份核实的 active / primary 时段；未核实的机场不自动进入迁场玩法。
- 已知迁场使用显式转换记录，通用关闭检测只作为补充；同一历史事项只生成一次。
- 强制关闭前阻止推进季度；可选的新枢纽启用允许迁移或留在仍运营的旧机场。
- 迁移保留航线 uid 和经营指标，转移机场投资价值、升级、折扣和机场关系，并结束不再适用的机场合同。
- 每条航线两端可指定同城或 250km 范围内、当年启用且与已派机型兼容的高置信度备降机场。
- 机场中断修正器同时影响需求和成本；备降计划与灾害韧性升级可单独缓解，两者叠加时效果最强。
- 时刻资产不在本轮实现。当前的季度容量点、连续服务强度和 AI 50% 背景拥堵不足以支撑可公平交易的独占资产；详见 `docs/airport-slot-feasibility.md`。

### 19.3 仍延后到后续阶段

- 最终人口压缩指数。
- 所有机场的 feeTier 和 capacityTier。
- 是否允许同一城市对多条机场组合航线。
- 时刻资产的离散时段、AI 竞价、价格发现和转售实现。
