# 中后期长局基线

本基线用于验证 40—80 条以上航线、80—120 架飞机时的操作密度、状态可读性和季度推进稳定性。它不是新的平衡策略，也不改变经济公式。

## 固定场景

夹具通过 `src/simulation/balance.js` 的激进策略和正式领域 API 生成，不直接拼装航线或绕过资金、机场、机型和人员规则。

```text
时代：era4 传奇时代
总部：伦敦
策略：aggressive
种子：long-game-baseline
季度：200
```

运行：

```bash
npm run long-game:baseline
npm run long-game:baseline -- --json
```

## 初始基线

固定场景应满足：

- 60—90 条航线；
- 80—120 架飞机；
- 至少 6 个分部；
- 每次季度推进后通过 `assertGameState`；
- 可序列化为当前存档格式并继续推进季度。

命令同时记录生成耗时、存档字节数，以及从同一状态克隆后推进 10 次季度的中位和最大耗时。耗时只用于本机前后对比，不作为跨机器硬阈值。

2026-07-11 的初始参考结果：

| 指标 | 结果 |
| --- | ---: |
| 航线 | 83 |
| 飞机 | 81（自有 57 / 租赁 24） |
| 分部 | 8 |
| 低客座率 / 无飞机航线 | 1 / 2 |
| 4 季内到期租约 / 2 年内退役自有机 | 1 / 1 |
| 存档大小 | 169,127 bytes |
| 生成耗时 | 约 4—6 秒 |
| 克隆后季度推进中位耗时 | 约 15 ms |

该自然经营状态没有刻意制造亏损、停飞或空闲飞机；后续 UI 验收会在其副本上注入可解释的诊断状态，不改变这份自然基线。

## 航线诊断验收

航线管理使用统一的领域诊断结果区分需关注、亏损、低客座、无飞机、停飞和合同航线。新开或刚复飞的航线在首次结算前不会被误判为亏损或低客座；本季度调价、换机后继续使用上一份已结算指标，直到下季度产生新结果。

Playwright 会在固定长局副本中注入一条亏损、一条低客座、一条停飞和一条合同航线，并保留自然产生的无飞机航线。桌面表格与手机卡片均验证筛选计数、每页数量、状态徽标、焦点保留和横向溢出。

## 机队与季度风险验收

机队更新计划按正式规则计算每架飞机距租约到期或 25 年退役的季度数，并汇总下季、4 季和 8 季内的离场飞机、受影响航线与需替代有效座位。交付中、在飞、空闲和租赁状态使用同一份领域快照，不依赖界面字符串反推。

下一季度预览会克隆当前状态，只应用确定的飞机交付与离场、分部完工，再重算静态航线与公司收支。它不会推进原状态或消费随机序列，也不会把随机新闻、故障、证券、子公司和合同实际结算伪装成确定结果。

双视口长局 E2E 在自然基线副本中加入一架下季退役飞机、一架下季到期租赁飞机和一架下季交付飞机，连续验证航线诊断、季度预览和机队更新筛选，并保存三组界面截图用于视觉复核。

批量管理验收会再加入一架自有空闲机和一架租赁空闲机，验证选择、影响预览、二次确认和整批执行。航线批量停飞、关闭、调价与换机的延后理由和立项条件记录在 `docs/batch-management-assessment.md`。

## 后续验收

后续航线诊断、机队更新和批量管理必须使用该夹具验证：

- 桌面和手机可以在密集状态下找到亏损、低客座率、停飞和无飞机航线；
- 可以识别临近退役、即将到期、交付中和空闲飞机；
- 筛选、分页或选择状态不会因弹窗切换而意外丢失；
- 批量操作必须先展示影响航线、预计现金和不可执行原因；
- `standalone.html` 仍可离线运行，且玩家主包不引入模拟器代码。
