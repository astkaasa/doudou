# 机场时刻资产可行性评估

## 结论

本轮不实现可交易、竞拍或转售的机场时刻资产。当前系统的最小时间单位是季度，机场容量是季度容量点，航线只有服务强度，没有可唯一占用的日期、时段或班次。在这一基础上直接把“时刻”做成可买卖资产，价格与稀缺性无法由实际供需得出，也无法让 AI 公平参与。

因此，阶段 5 保留已经可验证的容量、合同、机场关系、历史迁场和备降韧性玩法；时刻交易进入“有明确前置条件的延后项”，不使用随机价格或纯 UI 包装假装实现。

## 现有能力盘点

| 维度 | 现有模型 | 对时刻交易的支撑 |
| --- | --- | --- |
| 时间 | 1 回合 = 1 季度 | 不能表达早晚高峰、星期或单一时段 |
| 航线 | 城市对 + 机场端点 + 服务强度 | 没有班次及其占用的稀缺资源 |
| 容量 | 机场季度容量点，玩家按完整值、AI 按 50% 折算 | 可表达拥堵，不能标识可转移的具体权利 |
| 稀缺性 | 超容量增加成本、降低需求并提高开线压力 | 有后果，但没有新增、续期、收回与买方分配规则 |
| 机场交互 | 机场投资、升级、关系、合同与机会池 | 可作为时刻获取的前置，但不是价格发现机制 |
| AI | 开线并以 50% 容量影响背景拥堵 | 不会报价、放弃、转售或根据预期利润竞拍 |
| 历史 | 机场启停和迁场事项 | 尚无历史协调机制、时刻政策或时段级数据 |
| 存档 | 稳定机场 ID、航线 uid 和显式迁移 | 可以扩展，但需先定义时刻资产的稳定身份与生命周期 |

## 不通过的实现方案

以下方案在当前模型中不进入实现：

- 按机场等级生成一个“时刻价”，购买后只提高需求或容量。这与现有升级和关系奖励重复。
- 用随机拍卖事件代替供需。没有 AI 报价和机场容量减少时，竞拍只是一次性支出弹窗。
- 把每条航线的服务强度直接命名为时刻数。服务强度是连续经济倍率，不具备独占、到期、转让或机场时段冲突语义。
- 只让玩家买卖时刻，AI 仍作为固定拥堵值。这会让玩家承担成本，但没有可竞争对手或可验证的市场。

## 进入实现的前置条件

时刻资产只在以下条件同时满足后立项：

1. 定义抽象但可离散的时段，例如每座机场的“高峰 / 平峰”两类，且不引入真实日期、小时或昼夜状态。
2. 每个资产具有稳定 ID、机场 ID、时段、容量占用、开始季度、到期季度和持有者。
3. 新增、续期、收回、转售和迁场转移有唯一的领域规则，不与航线、合同或机场投资重复计算。
4. AI 能根据预计航线利润、机场关系、已有网络和现金压力决定报价、保留或卖出。
5. 价格由机场容量、当期需求、合同流量和多个买方的实际决策形成，并设置价格上限与防囤积规则。
6. UI 能同时说明价格、权利、占用航线、到期、迁场后果和转售预估，手机视口不依赖大表格。
7. 存档迁移能处理重复 ID、不存在的机场、过期资产、迁场中资产和旧航线无时刻的回退。
8. 至少通过领域、存档、AI 竞价、双视口 UI 和五总部平衡矩阵验收。

## 候选最小垂直切片

若后续立项，先只在伦敦和纽约验证一个抽象切片：

- 每座机场仅有高峰与平峰两类时段。
- 时刻与具体航线 uid 绑定，一条航线每端最多占用一个。
- 机场关系和开发合同影响首发获取，但不直接生成无限资产。
- AI 与玩家使用相同报价函数和现金约束。
- 迁场时显式决定转移、放弃或重新申请。

该切片仍需单独设计和平衡验收，不纳入当前机场城市系统 Goal 的完成条件。
